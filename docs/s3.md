# Amazon S3 Storage Providers - Complete Documentation

## Overview

Rclone supports Amazon S3 and numerous S3-compatible storage providers. The documentation provides comprehensive configuration and usage guidance for working with S3 storage systems.

## Supported S3 Providers

The S3 backend works with 70+ providers including:

- **Major Cloud Providers**: AWS S3, Google Cloud Storage, Microsoft Azure, Alibaba Cloud OSS
- **Object Storage Services**: Wasabi, Linode, DigitalOcean Spaces, Backblaze B2
- **Enterprise Solutions**: IBM COS, Huawei OBS, Tencent COS, Oracle Object Storage
- **Self-Hosted/Alternative**: Minio, Ceph, SeaweedFS, Storj
- **Regional Providers**: Scaleway, OVHcloud, Hetzner, various others across Asia, Europe, Americas

## Basic Usage

Paths follow the format: `remote:bucket` or `remote:bucket/path/to/dir`

Common operations:
- List buckets: `rclone lsd remote:`
- Create bucket: `rclone mkdir remote:bucket`
- List contents: `rclone ls remote:bucket`
- Sync directory: `rclone sync /home/local/directory remote:bucket`

## Configuration Process

Configuration begins with `rclone config`, which guides users through:

1. **Provider Selection**: Choose from AWS, Ceph, DigitalOcean, Dreamhost, IBM COS, Minio, Wasabi, or other S3-compatible providers
2. **Credentials**: Enter AWS Access Key ID and Secret Access Key
3. **Region Selection**: Choose appropriate region (or leave blank for clones)
4. **Optional Settings**:
   - Server-side encryption
   - Storage class selection
   - ACL configuration
   - Location constraint

## Authentication Methods

Rclone supports multiple authentication approaches:

- **Direct Configuration**: Access key ID and secret access key in config file
- **Environment Variables**: AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_SESSION_TOKEN
- **Named Profiles**: Standard AWS credential files (~/.aws/credentials)
- **IAM Roles**: EC2, ECS, and EKS metadata
- **Process Credentials**: External program-based authentication

### Cross-Account Access (Assume Role)

Configuration parameters for accessing resources in different AWS accounts:
- `role_arn`: The IAM role ARN to assume
- `role_session_name`: Optional session name
- `role_session_duration`: Optional credential duration
- `role_external_id`: Optional security identifier

## Modification Times and Hashes

**Metadata Storage**: Modification times stored as `X-Amz-Meta-Mtime` (floating point since epoch)

**Hash Implementation**:
- Small single-part uploads use ETag as MD5 checksum
- Multipart and server-encrypted uploads store `X-Amz-Meta-Md5chksum` as base64-encoded MD5
- Additional HEAD requests required to read metadata

## Cost Reduction Strategies

### Avoiding HEAD Requests

**Size-Only Comparison** (`--size-only`):
Only checks file size, no additional transactions, but misses changes where size remains constant.

**Checksum Verification** (`--checksum`):
Checks size and MD5, no extra transactions, most accurate change detection. "Recommended flag to use for maximum efficiency" when syncing between S3 locations.

**Server Modification Time** (`--update --use-server-modtime`):
Uses no extra API calls, though modification time becomes upload time rather than actual file modification.

**Directory Listing Optimization** (`--fast-list`):
Reads object information into memory using fewer API calls (one per 1000 objects). Trades API transactions for RAM usage (~1KB per object).

**Targeted Copying** (`--max-age` with `--no-traverse`):
Copies only recent files directly without full directory listings, suitable for "top-up" syncs.

### HEAD Request Reduction After PUT

The `--s3-no-head` flag disables automatic object verification after upload, reducing costs but increasing undetected failure risk.

## Performance Optimization

### Server-Side Copy

Copy objects between S3 buckets in the same region without data transfer by using the same remote for source and destination: `rclone copy s3:source-bucket s3:destination-bucket`

### Increasing API Request Rate

Adjust parallelism using `--transfers` and `--checkers` flags. Example with high concurrency:
`rclone sync --transfers 200 --checkers 200 --checksum s3:source-bucket s3:destination-bucket`

Optimal values depend on provider capabilities and require experimentation.

## Data Integrity

Rclone implements comprehensive verification:

- **HTTP Protection**: X-Amz-Content-Sha256 or Content-MD5 headers guard body integrity
- **Signature Protection**: Authorization header protects HTTP headers
- **HTTPS Encryption**: All communication encrypted and error-protected
- **Single-Part Uploads**: Content-MD5 hash verification on receipt
- **Multipart Uploads**: Each part protected with X-Amz-Content-Sha256 and Content-MD5
- **Downloads**: MD5 verification against ETag or X-Amz-Meta-Md5chksum metadata
- **Post-Upload Verification**: HEAD requests confirm successful arrival (can disable with `--s3-no-head`)
- **Additional Assurance**: `rclone check` verifies hashes locally vs remote; `rclone check --download` provides maximum verification

## Versioning

When bucket versioning is enabled:

- New file uploads create new versions
- Deletions mark old versions as hidden but retain them
- View all versions with `--s3-versions` flag
- View bucket state at specific time with `--s3-version-at` flag
- Remove old versions: `rclone backend cleanup-hidden remote:bucket`
- Purge deletes current and old versions, then removes bucket

**Naming Convention**: Versions include timestamps between filename and extension (e.g., `file-v2023-07-17-161032-000.txt`)

## Cleanup Operations

**Multipart Upload Cleanup** (`rclone cleanup s3:bucket`):
Removes pending uploads older than 24 hours. Use `--interactive` or `--dry-run` for preview. Control expiry with `--s3-max-age` flag (e.g., `rclone backend cleanup s3:bucket -o max-age=1h`).

View pending uploads: `rclone backend list-multipart-uploads s3:bucket`

## Restricted Filename Characters

S3 accepts valid UTF-8 keys. Rclone replaces:
- Invalid UTF-8 bytes
- NUL character (0x00) → ␀
- Forward slash (0x2F) → ／

Additionally encodes:
- Single dot (.) → ．
- Double dots (..) → ．．

**Critical Limitation**: Consecutive forward slashes (//) are not supported. Rclone normalizes `a//b` to `a/b`, causing "object not found" errors. Workaround: avoid `//` in S3 object keys.

## Multipart Uploads

**Upload Cutoff**: Files above `--s3-upload-cutoff` (default enables multipart for large files) split into chunks.

**Configuration**:
- `--s3-chunk-size`: Size of each part
- `--s3-upload-concurrency`: Number of concurrent chunks

**Memory Usage**: `--transfers` × `--s3-upload-concurrency` × `--s3-chunk-size`

**Performance**: Single vs multipart speed depends on latency; increase concurrency and chunk size for throughput improvements (e.g., 8 concurrent transfers, 16MB chunks).

**Maximum Size**: 5 GiB file limit, minimum 0 (always multipart).

## Buckets and Regions

List buckets from any region, but access bucket content only from the region where created. Cross-region access returns: "incorrect region, the bucket is not in 'XXX' region".

## S3 Permissions

**Minimum Requirements**:
- ListBucket
- DeleteObject
- GetObject
- PutObject
- PutObjectACL
- CreateBucket (unless using `--s3-no-check-bucket`)
- ListAllMyBuckets (for `lsd` command)

Example IAM policy provided in documentation for bucket creation and syncing.

## Key Management System (KMS)

Configure `server_side_encryption = aws:kms` when using KMS encryption to prevent checksum errors on small objects.

## Glacier and Glacier Deep Archive

Objects in glacier storage classes must be restored before access. Error message: "Object in GLACIER, restore first". Use AWS restore process before rclone access. Note: rclone uses S3 API only; cannot directly access Glacier Vaults.

## Object-Lock Enabled Buckets

AWS requires Content-MD5 header for bucket uploads with default retention. Solution: use `--s3-upload-cutoff 0` to force multipart uploads on all files.

## Standard Configuration Options

### Provider Selection (`--s3-provider`)

Options include: AWS, Alibaba, ArvanCloud, BizflyCloud, Ceph, ChinaMobile, Cloudflare, Cubbit, DigitalOcean, Dreamhost, Exaba, FileLu, FlashBlade, GCS, Hetzner, HuaweiOBS, IBMCOS, IDrive, Intercolo, IONOS, Leviia, Liara, Linode, LyveCloud, Magalu, Mega, Minio, Netease, Outscale, OVHcloud, Petabox, Qiniu, Rabata, RackCorp, Rclone, Scaleway, SeaweedFS, Selectel, Servercore, SpectraLogic, StackPath, Storj, Synology, TencentCOS, Wasabi, Zata, Other

### Environment Authentication (`--s3-env-auth`)

Boolean flag (default: false) enabling runtime AWS credentials from environment variables or EC2/ECS metadata.

### Region Configuration (`--s3-region`)

Supports 50+ AWS regions and region-specific endpoints for alternative providers. Leave blank for S3 clones without region requirements.

### Endpoint Configuration (`--s3-endpoint`)

Required for S3-compatible clones. Extensive list of provider-specific endpoints available in documentation (Alibaba, ArvanCloud, Ceph, DigitalOcean, Dreamhost, Exaba, FileLu, GCS, Hetzner, HuaweiOBS, IBM COS, and others).

### Location Constraint (`--s3-location-constraint`)

Must match region when creating buckets. Leave blank if uncertain. Supports regions matching provider endpoints.
