# Backblaze B2 on Rclone - Complete Documentation

## Overview
Backblaze B2 represents a cloud storage solution integrated with rclone for file synchronization and management. The system uses path specifications in the format `remote:bucket` with optional subdirectories.

## Configuration Process

Initial setup requires running `rclone config` to enter an interactive configuration wizard. Users must provide either:
- Account ID (short hex number) and Master Application Key (long hex number), OR
- An Application Key (recommended approach)

Basic configuration example establishes a remote named "remote" with authentication credentials stored for subsequent operations.

## Key Features

### Application Keys
"B2 supports multiple Application Keys for different access permission to B2 Buckets." Users implementing this feature must use rclone version 1.43 or later and must input the applicationKeyId as the account parameter rather than the master Account ID.

### Fast List Support
The `--fast-list` flag enables reduced transaction counts in exchange for increased memory consumption, optimizing performance for specific use cases.

### Modification Time Storage
Modification times store as metadata using `X-Bz-Info-src_last_modified_millis` (milliseconds since 1970-01-01). When modification updates occur, new object versions are created.

### Restricted Characters
Beyond default restrictions, the backslash character (0x5C) becomes ďĽĽ. Invalid UTF-8 bytes also receive replacement. Note: B2 permitted backslash characters starting May 2020, but rclone maintains its encoding to prevent unnecessary file retransfers.

### SHA1 Checksums
"The SHA1 checksums of the files are checked on upload and download and will be used in the syncing process." Large files exceeding the upload cutoff store SHA1 as `X-Bz-Info-large_file_sha1`.

### Transfer Optimization
"Backblaze recommends that you do lots of transfers simultaneously for maximum speed." Testing suggests optimal settings around `--transfers 32`, though hardware specifications and file sizes influence ideal configurations. Default `--transfers 4` falls below B2 recommendations.

## File Versioning

B2 maintains historical file versions by default. Operations create new versions rather than overwriting. The `--b2-versions` flag displays old versions with UTC timestamp suffixes. Users may implement "hard delete" with `--b2-hard-delete` to permanently remove files instead of hiding them.

Version management commands include:
- `rclone cleanup remote:bucket` - removes old versions and unfinished uploads
- `rclone backend cleanup-hidden` - removes only old hidden versions
- `--b2-version-at` flag - shows bucket state at specified points in time

## Shared Link Generation

The `rclone link` command generates shareable URLs for private bucket access:
- File links: `https://f002.backblazeb2.com/file/bucket/path/to/file.txt?Authorization=xxxxxxxx`
- Directory links provide authorization tokens applicable to subdirectories

## Standard Configuration Options

| Option | Details |
|--------|---------|
| `--b2-account` | Account ID or Application Key ID (required) |
| `--b2-key` | Application Key (required) |
| `--b2-hard-delete` | Boolean for permanent file deletion (default: false) |

## Advanced Options

### Upload Configuration
- `--b2-upload-cutoff`: Switches to chunked uploads (default: 200Mi, maximum: 4.657 GiB)
- `--b2-chunk-size`: Individual chunk size (default: 96Mi, minimum: 5MB)
- `--b2-upload-concurrency`: Concurrent chunk uploads (default: 4)

### Copy Operations
- `--b2-copy-cutoff`: Multipart copy threshold (default: 4Gi, range: 0 to 4.6 GiB)

### Security Features
- `--b2-sse-customer-algorithm`: Server-side encryption (supports "AES256")
- `--b2-sse-customer-key`: UTF-8 encoded encryption key
- `--b2-sse-customer-key-base64`: Base64 encoded encryption key alternative

### Additional Settings
- `--b2-test-mode`: Debug flag supporting fail_some_uploads, expire_some_account_authorization_tokens, force_cap_exceeded
- `--b2-download-url`: Custom CDN endpoint for downloads
- `--b2-download-auth-duration`: Public link token expiration (default: 1 week, range: 1 second to 7 days)
- `--b2-lifecycle`: Days to retain deleted files (0 = default B2 behavior)

## Backend Commands

### Lifecycle Management
The `rclone backend lifecycle` command reads/modifies bucket lifecycle rules:
- `daysFromHidingToDeleting`: Deletion delay after hiding
- `daysFromUploadingToHiding`: Hide delay after upload
- `daysFromStartingToCancelingUnfinishedLargeFiles`: Unfinished upload cancellation period

### Cleanup Operations
- `rclone backend cleanup`: Removes unfinished uploads older than 24 hours (configurable via `-o max-age`)
- `rclone backend cleanup-hidden`: Removes old hidden file versions

## Data Usage Metrics

Standard copy operations generate 4 baseline API requests. The `b2_list_file_names` request repeats once per 1,000 files in the remote path. File uploads generate 2 requests per file; chunked uploads require 2 additional requests per chunk plus start/finish requests.

## Limitations

"rclone about is not supported by the B2 backend." This prevents free space determination for rclone mounts and blocks policy mfs (most free space) usage within union remotes.
